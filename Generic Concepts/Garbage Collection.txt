Now that you know about memory lingo, its about time you know about how memory works. And this concept is not specific for any language but rather
for all languages.

There is a concept of a stack, heap, text-segment, and global section in memory. Each program when loaded into the RAM is assigned these things,
basically the OS (operating system) assigns a region in RAM for each program to use. Let's talk about each section.


(i) Text-segment: this section in memory contains the actual code, of course it doesn't contain words but binary equivalent of the code you write.

(ii) Global Section: it contains the global/static variables

(iii) Stack: this is where all the function calls are are resolved, all the variables declared in a function are stored here. It also stores other things
             but that is the main thing which is stored here. This memory is static in size and is fixed during runtime.

(iv) Heap: This section is used when the program runs out of memory in stack (Remember StackOverflow Errors), variables stored here persist even after
           the command is shifted from the calling function. Or you just use this (if you can) to store variables that you want to use later, closure variables
           are also stored here (See notes). This type of memory is dynamic in size and can be changed during runtime.


The garbage collector or the GC of any language does only one thing, when variables are no longer in use it deletes it from memory, so that other
variables which are in use may use the freed memory.


Memory Management in the Stack

When a function is called or a line of code is executed, the variables which the line of code declares is stored in the Stack.
If a function is called, there is something called a Stack-Frame which is created, fixed in size.

Like say you have the following function in python:


def do_something():
    a = 1
    b = 2
    print(f"a + b: {a + b}")
    

When the above function is called it will create its own stack-frame in the stack section of the memory which will look like this:

_________________________
|         STACK         |
|_______________________|                       
|do_something()         |
| a = 1  b = 2          | <--------- a stack-frame holding two variables         
|_______________________|                       

When a function ends all the memory used in the stack frame is freed and given back to the program automatically.         


Memory Management in the Heap

This is where the job of the garbage collector begins, you see unlike the stack section of the memory, the heap doesn't automatically
gets its memory freed if the variables stored in it are no longer in use. Its the job of the garbage collector to clear memory in the heap,
if the language you are using doesn't have a garbage collector like C or C++ then it is the job of the programmer to do so.

But humans can make mistakes, and so languages where the programmer manages memory manually are usually considered unsafe, but really fast.
That is because the garbage collector induces additional runtime overhead in a program which slows down operations.
